# 法定相続分 計算Webアプリ 要件定義書

最終更新：2026-04-30

---

## 1. プロジェクト概要

### 1.1 ゴール
**「相続関係図を入力するだけで、法定相続分が自動で計算・表示されるWebアプリ」** を一般の方向けに提供する。

### 1.2 提供価値
- 一般の方が自分の相続ケースで「誰が、いくら（何分の何）もらえるのか」を直感的に把握できる
- 入力した関係図と計算結果が**ひと目で対応**して見える
- 専門用語を知らなくても操作できる

### 1.3 ノンゴール
- 遺産分割協議書などの**法的書類の自動生成は対象外**（あくまで概算把握ツール）
- 相続税の計算は対象外（別物）
- ツールの結果は参考値であり、最終判断は専門家に確認する旨を明示

---

## 2. 利用者像（ペルソナ）

### 主ペルソナ：相続に直面した一般の方
- 法律知識は乏しい
- 専門用語（被相続人、直系尊属、代襲相続など）を知らない前提
- 自分や家族の取り分の**概算**を知りたい
- スマホ・PC どちらからでもアクセスする

### 副ペルソナ：法律の勉強中の人
- 試験勉強や独学で法定相続分のパターン演習に使う

---

## 3. 利用シーン

1. 親が亡くなり、自分・きょうだい・配偶者の取り分を知りたい
2. 祖父が亡くなり、すでに父も他界しているケース（代襲）の自分の取り分を知りたい
3. 子がいない夫婦の片方が亡くなり、配偶者と兄弟姉妹で分けるケース
4. 養子がいる、半血兄弟がいる、放棄者がいるなど **複雑なケースの計算結果を確認したい**

---

## 4. 機能要件

### 4.1 入力モード（切り替え可能）

#### モードA：ウィザード入力（初心者向け・デフォルト）
質問に答えていくだけで関係図が組み上がる。
- 「亡くなった方（被相続人）から見た関係を教えてください」
- 「配偶者はいますか？」（はい／いいえ／離婚済み／死別）
- 「子はいますか？」「何人ですか？」
- 子ごとに「現在も生存していますか？」「（亡くなっている場合）その方に子はいますか？」（代襲判定）
- 「相続放棄した方はいますか？」
- 子がいない場合は「親（直系尊属）はご存命ですか？」へ
- 親もいない場合は「兄弟姉妹はいますか？」へ
- 各分岐で**専門用語は使わず、平易な日本語**で質問

#### モードB：ビジュアル編集（自由度重視）
キャンバス上で人物ノードを配置・関係線をつないで関係図を作る。
- 人物ノード追加（「本人（被相続人）」「配偶者」「子」「親」「兄弟姉妹」「孫」など）
- 各ノードに「生存／死亡」「相続放棄」「相続欠格」「廃除」「養子（普通／特別）」「半血／全血（兄弟の場合）」フラグ
- 親子関係・婚姻関係を線でつなぐ
- ドラッグで自由配置／自動レイアウトボタンあり

#### モード切替
- 任意のタイミングで A↔B を切り替え可能
- ウィザードで入れた内容はビジュアルにそのまま反映される（同一の内部データモデル）
- ビジュアルで編集後にウィザードに戻ると、対応する設問の回答が更新されている

### 4.2 計算ロジック（コア機能）

民法に基づく法定相続分を以下の全ケースに対応して計算する。

| ケース | 内容 | 対応優先度 |
|---|---|---|
| **第1順位** | 配偶者＋子（子のみ含む） | 必須 |
| **第2順位** | 配偶者＋直系尊属 | 必須 |
| **第3順位** | 配偶者＋兄弟姉妹 | 必須 |
| **配偶者のみ** | 配偶者が全部 | 必須 |
| **代襲相続** | 子・兄弟姉妹の代襲（兄弟姉妹は1代限り） | 必須 |
| **半血兄弟姉妹** | 全血の1/2 | 必須 |
| **養子** | 普通養子は実親・養親の両方／特別養子は養親のみ | 必須 |
| **相続放棄** | 最初から相続人でないとみなす（代襲なし、次順位へ移る場合あり） | 必須 |
| **相続欠格・廃除** | 当人は失権するが**代襲原因になる** | 必須 |
| **数次相続** | 相続中の死亡発生に対応（被相続人を切り替えて連続計算） | 必須 |
| **同時死亡の推定** | 互いに相続しない／代襲はある | 必須 |

> 注：非嫡出子の取扱いは**スコープ外**（現行法では嫡出子と同じため、特別な分岐を設けない）。

#### 計算結果の表示
- 各相続人の **法定相続分（分数）** を一覧表示（例：配偶者 1/2、子A 1/4、子B 1/4）
- **関係図に直接書き込む形式**でも表示（ノード横に「1/4」と表示）
- 計算根拠の説明（「配偶者は子と相続するので 1/2、子は残り 1/2 を均等に分けるので 1/4 ずつ」）
- 該当する民法条文（900条、901条、887条など）を参考表示

### 4.3 出力・エクスポート
- 計算結果と関係図を **PDF 出力** できる
- 関係図を **画像（PNG）** で保存できる
- URL 共有（入力内容をクエリ／ハッシュにエンコードして共有可、サーバー保存なし）

### 4.4 データ保存
- **サーバーには一切保存しない**（プライバシー配慮、家族構成は機微情報）
- ブラウザのローカルストレージに保存（自分のPCにのみ残る）
- 「保存」「読み込み」「クリア」操作

### 4.5 補助機能
- **サンプルケース読み込み**（典型パターン10例ほど）
- **用語ヘルプ**（？マークホバーで「代襲相続とは？」を表示）
- **免責表示**（「本ツールの結果は参考値です。最終判断は専門家にご確認ください」を常時表示）

## 5. 非機能要件

### 5.1 プラットフォーム
- **モダンブラウザ対応**：Chrome / Edge / Safari / Firefox の最新版
- **レスポンシブ**：スマホ／タブレット／PC で利用可
- ビジュアル編集モードは PC 推奨、スマホではウィザード優先

### 5.2 パフォーマンス
- 入力反映から計算結果表示まで **1秒以内**
- 関係者100人規模でも計算可（実用上は20人程度を想定）

### 5.3 セキュリティ／プライバシー
- 入力データを**サーバーに送信しない**（クライアント完結）
- アクセス解析等は最小限、もしくは不使用
- 「個人情報を扱わない仕様」を明記

### 5.4 アクセシビリティ
- キーボード操作可能
- 色覚多様性に配慮した配色
- 最低限の WCAG AA 相当を目指す

### 5.5 多言語
- 日本語のみ（民法準拠なので国際化は対象外）

## 7. データモデル（暫定）

```ts
type Person = {
  id: string;
  name?: string;          // 任意（「子A」のような匿名表記もOK）
  role: 'decedent' | 'spouse' | 'child' | 'parent' | 'sibling' | 'grandchild' | ...;
  status: 'alive' | 'deceased' | 'unknown';
  flags: {
    renounced?: boolean;       // 相続放棄
    disqualified?: boolean;    // 欠格
    excluded?: boolean;        // 廃除
    adopted?: 'normal' | 'special' | null; // 養子
    halfBlood?: boolean;       // 半血（兄弟姉妹のみ）
    illegitimate?: boolean;    // 非嫡出
  };
  parentIds?: string[];   // 親へのリンク（代襲・直系判定に使う）
  spouseId?: string;
};

type Case = {
  decedentId: string;
  persons: Person[];
  options: {
    applyOldLawForIllegitimate?: boolean; // 旧法適用（非嫡出1/2）
  };
};
```

## 8. 技術スタック（暫定案）

- **言語**：TypeScript
- **フレームワーク**：Next.js (App Router) または Vite + React
  - サーバー不要なので静的書き出し可能なものを選択
- **UI**：Tailwind CSS + shadcn/ui
- **ビジュアル編集**：React Flow（ノード・エッジ編集に最適）
- **PDF 出力**：jsPDF または print-to-PDF
- **ホスティング**：Vercel / GitHub Pages（静的配信）
- **計算ロジック**：純粋関数として独立モジュール化（テスタブル）

→ **次フェーズで決定**。優先度は「計算ロジック」「ビジュアル編集」の実装容易性。

---

## 9. テスト方針

- 計算ロジックは **ユニットテスト必須**（民法の各ケースを網羅）
- テストケースは判例・教科書例を引用しトレーサビリティを確保
- ゴールデンファイル方式（入力JSON → 期待出力 JSON）で回帰防止

## 11. 開発進行方式：PDCA（Cはユーザー）

本プロジェクトは PDCA で短サイクルに回す。**Check はユーザーが実機で確認**、それ以外（P/D/A）は開発側が回す。

| フェーズ | 担当 | 内容 |
|---|---|---|
| **Plan** | 開発側 | 各ラウンドのスコープを宣言（小さく区切る） |
| **Do** | 開発側 | 動くものを作る（毎ラウンドでブラウザで開ける状態にする） |
| **Check** | **ユーザー** | 実機で触り、計算結果・UX をレビューしてフィードバック |
| **Action** | 開発側 | フィードバックを反映、次ラウンドの Plan へ |
